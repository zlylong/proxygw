# ProxyGW 上线前稳定性与安全审计（2026-04-16）

审计目标：围绕“**稳定性** + **安全性**”进行发布前快速审计，并优先修复可直接落地的问题。  
审计范围：`backend/nodes_routes.go`（节点导入、批量探测路径）。

---

## 本轮关键结论

### 1) 已修复：节点导入链路的错误吞噬问题（稳定性/安全性）
- `vmess://` 导入此前存在多个未处理错误（Base64 解码、JSON 解析、DB 写入、端口合法性），可能导致接口返回不一致或脏数据写入。
- `vless://` 导入此前同样缺失核心校验与 DB 错误处理。
- 现已补齐：
  - 输入有效性校验（地址、UUID、端口范围）；
  - 解析失败与序列化失败的明确 HTTP 错误返回；
  - DB 写入失败即时返回并停止后续流程。

### 2) 已修复：重复 `rows.Err()` 检查与 ping 接口异常响应缺失（稳定性）
- `/nodes` 与 `/nodes/ping` 中存在重复 `rows.Err()` 检查，增加维护噪音。
- `/nodes/ping` 在扫描出错时仅日志打印、不返回客户端错误，易造成“前端成功、后端部分失败”的观测偏差。
- 现已统一为单次检查并返回结构化错误响应。

---

## 风险评估（修复后）

- 节点导入接口对异常输入的抗性提升，减少错误配置进入运行面。  
- 后端失败路径可观测性更好，降低“伪成功”状态造成的发布风险。  
- 对外行为更确定，便于发布窗口内快速定位问题。

---

## 上线前建议（未在本次改动中落地）

1. 为 `/nodes/import` 增加速率限制与审计日志（操作者、来源 IP、节点数量）。  
2. 将节点导入解析拆分为纯函数并补充表驱动单测（含恶意/畸形链接样例）。  
3. 对全局数据库操作继续推进“错误必须处理”基线，避免 silent failure。
